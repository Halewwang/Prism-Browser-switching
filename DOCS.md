# Prism 功能优化与使用指南

## 1. 来源 App 识别与显示优化

我们对来源 App 的识别机制进行了全面升级，确保在历史记录和弹窗中能够准确显示唤起链接的应用程序。

### 支持的应用程序
目前已内置以下常用办公与通讯软件的标准化图标与名称支持：
- **微信 (WeChat)**
- **钉钉 (DingTalk)**
- **飞书 (Lark/Feishu)**
- **Slack**
- **Microsoft Teams**
- **Telegram**
- **QQ**

### 技术实现原理
- **识别机制**：通过 macOS 系统事件 (`osascript`) 精准获取当前前台唤起的应用程序进程名称。
- **标准化映射**：建立 `SourceAppNormalization` 层，将系统进程名（如 `DingTalk`）自动映射为用户友好的名称（`钉钉`）并匹配高清 SVG 图标。
- **一致性保证**：历史记录 (`History`) 与选择弹窗 (`Popup`) 共用同一套数据源，确保视觉体验高度统一。

## 2. 路由规则引擎重构

规则匹配逻辑已根据用户反馈进行了核心算法重构，大幅提升了准确性和灵活性。

### 新版匹配逻辑（优先级从高到低）

1.  **URL 规则（最高优先级）**
    *   **描述**：针对特定网址的匹配。
    *   **适用场景**：公司内网、特定会议链接（Zoom/Teams）、开发环境（localhost）。
    *   **示例**：
        *   规则：包含 `figma.com` -> 打开 `Arc`
        *   规则：包含 `localhost` -> 打开 `Chrome`
    *   **优势**：无论从哪个 App 点击，只要网址符合，立即生效。

2.  **来源应用规则（次级优先级）**
    *   **描述**：针对特定来源 App 的匹配。
    *   **适用场景**：希望“工作软件里的链接都用工作浏览器打开”。
    *   **示例**：
        *   规则：来源包含 `Slack` -> 打开 `Chrome`
        *   规则：来源包含 `微信` -> 打开 `Safari`
    *   **改进点**：现在只有在**未命中 URL 规则**的情况下，才会检查来源规则。避免了“在 Slack 里点 Zoom 链接却错误地用 Chrome 打开（而不是 Zoom App）”的问题。

## 3. 测试与验证

建议用户在更新后进行以下场景测试：

1.  **基础测试**：从 Slack/钉钉 点击一个普通链接（如 `baidu.com`），确认弹窗/历史记录中显示的来源图标是否正确。
2.  **规则测试**：
    *   设定一个 URL 规则（如 `github.com` -> Edge）。
    *   设定一个来源规则（如 `钉钉` -> Arc）。
    *   **预期结果**：在钉钉中点击 `github.com`，应**优先**触发 URL 规则（打开 Edge），而非来源规则。
